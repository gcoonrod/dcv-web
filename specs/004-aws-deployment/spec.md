# Feature Specification: AWS Multi-Origin Hosting & CI/CD

**Feature Branch**: `004-aws-deployment`
**Created**: 2026-03-07
**Status**: Draft
**Input**: Web Roadmap Phase 4 — Configure a unified CDN distribution for `apps.microcode.io` utilizing path-based routing to serve the Astro static site at `/dcv` and CLI binaries at `/dcv/releases/` from distinct storage origins.

## Clarifications

### Session 2026-03-07

- Q: How should directory-style URL requests (e.g., `/dcv/`) be resolved to an HTML document? → A: Use `build.format: 'file'` in Astro config — Astro emits flat `.html` files, simplifying path rewriting. A minimal CloudFront Function (viewer-request) is still required: OAC forces use of the S3 REST endpoint, which cannot auto-serve `index.html` for directory requests without a function. `build.format: 'file'` reduces the function's complexity (no nested directory-index rewriting needed) but does not eliminate the need for it.
- Q: What mechanism should restrict S3 bucket access to CloudFront only? → A: Origin Access Control (OAC) — modern signed-request mechanism; bucket policy grants access exclusively to the CloudFront distribution; no public bucket access.
- Q: What cache TTL policy should apply to binary artifacts at `/dcv/releases/*`? → A: Long TTL (1 year / immutable) — binaries are versioned by semver path and never overwritten; aggressive edge caching with no invalidation needed.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Astro Base Path Compatibility (Priority: P1)

As a developer, I want my local development environment to serve the site from the same sub-path as production (`/dcv`) so I don't accidentally write broken relative links that only fail after deployment.

**Why this priority**: This is foundational — without the correct base path, every asset reference and internal link generated by the build will be wrong in production, making the entire site non-functional at its CDN URL.

**Independent Test**: Configure the base path, run the local dev server, and verify the site is accessible at `http://localhost:4321/dcv/`. Then run a production build and inspect the output HTML to confirm all asset references include the `/dcv/` prefix.

**Acceptance Scenarios**:

1. **Given** the site configuration has a base path of `/dcv`, **When** a developer runs the local dev server, **Then** the site is accessible at `http://localhost:4321/dcv/` rather than the root.
2. **Given** the site is built for production, **When** the output HTML is inspected, **Then** all CSS, JS, and image asset references are prefixed with `/dcv/_astro/...`.
3. **Given** the base path is set, **When** any internal navigation link is rendered, **Then** the link href includes the `/dcv` prefix automatically.

---

### User Story 2 - Automated Deployment & Scoped Cache Invalidation (Priority: P1)

As the maintainer, I want to push a website update to the `main` branch and have the CI pipeline automatically deploy it, invalidating only the `/dcv/*` cache path so I don't accidentally purge cached assets for other applications sharing the same CDN distribution.

**Why this priority**: Manual deployments are error-prone and slow. Scoped cache invalidation is critical because this CDN distribution may serve other applications; a wildcard (`/*`) invalidation would impact unrelated apps and violate the shared-infrastructure contract.

**Independent Test**: Push a commit to `main`. Observe the CI pipeline complete successfully, confirm the updated files appear in the web S3 bucket, and verify a cache invalidation was created with the path `/dcv/*` (not `/*`).

**Acceptance Scenarios**:

1. **Given** a push event to the `main` branch, **When** the CI pipeline triggers, **Then** it checks out the code and produces a production build artifact.
2. **Given** a successful build, **When** the pipeline authenticates to the cloud provider, **Then** it uses short-lived, keyless credentials via a federated identity mechanism (OIDC) — no long-lived secrets stored in CI.
3. **Given** authenticated access, **When** the pipeline syncs files, **Then** it uploads all files in the build output to the web-specific storage bucket and removes any files that no longer exist in the build.
4. **Given** a successful sync, **When** the pipeline triggers a cache purge, **Then** the invalidation path is strictly `/dcv/*` and does not include paths for other applications.

---

### User Story 3 - Secure Binary Distribution via CDN (Priority: P1)

As a user running `dcv upgrade`, I want my CLI tool to download the latest signed binaries from the official CDN endpoint over HTTPS so I can trust the integrity and authenticity of what I'm installing.

**Why this priority**: The release distribution pipeline is a hard dependency for every `dcv upgrade` command executed by end users. If binaries are not accessible at the expected CDN path, the self-update feature is broken for all users.

**Independent Test**: After a release is published to the binary S3 bucket under a versioned path (e.g., `v3.2.1/`), make an HTTPS request to `https://apps.microcode.io/dcv/releases/v3.2.1/dcv-darwin-arm64` and verify a `200 OK` response with the binary content.

**Acceptance Scenarios**:

1. **Given** a release publishing script runs with a version and artifacts, **When** it syncs binaries to the storage backend, **Then** the binaries, checksum file (`SHA256SUMS.txt`), and signature file (`.sig`) are stored under a versioned folder path (e.g., `/v3.2.1/`).
2. **Given** the CDN is configured with path-based routing, **When** a request arrives for `/dcv/releases/v3.2.1/dcv-darwin-arm64`, **Then** the CDN routes it to the binary-specific storage bucket (not the website bucket) and returns the file.
3. **Given** the CDN serves the binary, **When** the user's CLI verifies the download, **Then** the SHA256 checksum matches `SHA256SUMS.txt`, confirming file integrity.

---

### Edge Cases

- What happens when a CDN request arrives for `/dcv/` (directory-style URL)? Resolved by the CloudFront Function on the `/dcv/*` behavior, which rewrites `/dcv/` → `/index.html` before forwarding to the S3 REST endpoint. `build.format: 'file'` reduces the function's logic (no nested directory-index rewriting) but the function itself is required because OAC disallows the S3 static website endpoint.
- What happens if the web S3 bucket sync fails mid-upload? The `--delete` flag means partial syncs could leave the bucket in an inconsistent state; the pipeline should treat the sync as atomic and surface errors immediately.
- What happens when a binary version is requested that does not exist? The CDN should return the storage bucket's `404` response rather than serving the website's `index.html` from the other origin.
- What happens if the OIDC token exchange fails? The pipeline must fail fast with a clear error before any file operations begin.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The site build configuration MUST use a base path of `/dcv` and `build.format: 'file'` so Astro emits flat `.html` files (e.g., `dcv.html`) rather than directory-scoped `index.html` files. A minimal CloudFront Function (viewer-request) MUST also be deployed on the `/dcv/*` behavior to handle index document resolution; this is required because Origin Access Control (OAC) mandates use of the S3 REST endpoint, which does not auto-serve `index.html` for directory requests.
- **FR-002**: The CI/CD pipeline MUST trigger automatically on every push to the `main` branch.
- **FR-003**: The pipeline MUST authenticate to the cloud provider using a keyless federated identity protocol (OIDC); storing long-lived access keys in CI secrets is prohibited.
- **FR-004**: The pipeline MUST sync the production build output to the web-specific storage bucket using a destructive sync (removing files deleted from the source).
- **FR-005**: The pipeline MUST execute a cache invalidation scoped strictly to the `/dcv/*` path after a successful sync.
- **FR-006**: The CDN MUST route `/dcv/releases/*` requests to the binary-specific storage bucket, taking priority over the broader `/dcv/*` behavior. The cache policy for this behavior MUST use a maximum TTL of 1 year (31,536,000 seconds); binary artifacts are immutable by versioned path and require no invalidation.
- **FR-007**: The CDN MUST route all other `/dcv/*` requests to the website storage bucket.
- **FR-008**: The CDN MUST serve all content over HTTPS only; HTTP access is prohibited.
- **FR-011**: Both storage buckets MUST block all public access directly. The CDN MUST access each bucket exclusively via Origin Access Control (OAC), with bucket policies that restrict reads to the specific CDN distribution only.
- **FR-009**: Directory-style requests (paths ending in `/`) under `/dcv/` MUST resolve to the correct HTML document without returning a `403` or `AccessDenied` error.
- **FR-010**: The deployment architecture MUST be documented in `DEPLOYMENT.md`, covering the dual-origin CDN routing model so future maintainers can understand the path routing without prior context.

### Key Entities

- **Website Storage Bucket**: The storage container holding the static site build output. Receives files from the CI sync step. All public access is blocked; content is accessible only via CDN using Origin Access Control (OAC).
- **Binary Storage Bucket**: The storage container holding versioned CLI release artifacts, checksums, and signatures. Updated by the release publishing script, not the CI deploy pipeline. All public access is blocked; content is accessible only via CDN using Origin Access Control (OAC).
- **CDN Distribution**: The single global content delivery network endpoint for `apps.microcode.io`. Routes requests to the correct origin based on path prefix using ordered behavior rules.
- **IAM Role (OIDC)**: The cloud identity assumed by the CI pipeline during deployment. Scoped with least-privilege permissions: write access to the website bucket and invalidation access on the CDN distribution only.
- **Release Artifact**: A versioned set of files (binary per target platform, `SHA256SUMS.txt`, `.sig`) published to the binary bucket under a path like `/v{semver}/`.

## Assumptions

- **CloudFront Index Document Handling**: `build.format: 'file'` is used in Astro config. Astro emits flat `.html` files (e.g., `dcv.html`) instead of `dcv/index.html`, which reduces the path-rewriting complexity needed at the CDN layer. However, a minimal CloudFront Function (viewer-request) is still required on the `/dcv/*` behavior: OAC mandates the S3 REST API endpoint, which does not auto-serve `index.html` for directory-style requests. The function strips the `/dcv` prefix and maps directory requests to `index.html` and extensionless paths to `.html`.
- **Shared CDN Distribution**: The CloudFront distribution for `apps.microcode.io` may serve other applications at other paths (e.g., `/another-app/*`). The deployment pipeline MUST NOT invalidate paths outside `/dcv/*`.
- **Binary Publishing is Out of Scope**: The `publish-release.sh` script that uploads artifacts to the binary bucket is owned by the CLI project (Phase 24). This spec covers only the CDN behavior and bucket configuration needed to serve those artifacts.
- **DNS Configuration**: DNS for `apps.microcode.io` pointing to the CDN distribution is assumed to be pre-configured or handled as an infrastructure prerequisite outside this feature's scope.
- **S3 Bucket Naming**: Actual bucket names are treated as environment-specific configuration values injected via CI secrets/variables (`WEB_BUCKET`, `DISTRIBUTION_ID`), not hardcoded.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: The DCV website is publicly accessible at `https://apps.microcode.io/dcv` within 5 minutes of a successful merge to `main`.
- **SC-002**: CLI binary artifacts are publicly downloadable from `https://apps.microcode.io/dcv/releases/<version>/<filename>` immediately after a release is published.
- **SC-003**: The CDN distribution requires zero load balancer or compute instances to perform path routing, resulting in $0.00/month in load balancer fees.
- **SC-004**: The CI pipeline completes a full build-sync-invalidate cycle in under 3 minutes for a standard site update.
- **SC-005**: Cache invalidations affect only the `/dcv/*` path, verified by confirming no invalidation paths outside this prefix are created during deployment.
- **SC-006**: 100% of pipeline runs authenticate using short-lived, automatically-rotated credentials with no long-lived keys stored in the CI environment.
- **SC-007**: Repeat downloads of any binary artifact (same version/platform) are served from CDN edge cache — not the storage origin — on all requests after the first, as verified by `X-Cache: Hit from cloudfront` response headers.
